investigative-pipeline

Build a queryable facts database from research documents, books, and media — for use with Claude Code

Install
$ git clone https://github.com/lammer-app/investigative-pipeline

Two pieces: a Python extraction script that turns source documents into a structured facts database, and a Claude Code skill that gives Claude Code direct access to query it.

  • Ingestion scripts included for EPUB, PDF, Kindle, articles, YouTube, and audio
  • Facts stored as JSONL — plain files, grep-friendly, no database to manage
  • Claude Code reads the facts database to answer research questions and generate drafts
  • No external dependencies — Python 3.11+ stdlib only

Overview

For research projects where evidence accumulates across many sources — books, court filings, depositions, transcripts, articles — and you need Claude Code to reason across all of it at once.

The extraction script reads source documents in markdown format and scores each sentence for factual density: named entities, dates, dollar amounts, attribution phrases. Sentences above the threshold become facts, each tagged with source, certainty, people, places, and organizations. They accumulate into a single facts.jsonl file across all sources.

The Claude Code skill points Claude at that database. From there, Claude can find every fact about a person across all sources, build timelines, surface connections, and generate annotated drafts with citations — all from a verified, source-linked corpus rather than from memory.

Ingestion

The extractor reads plain markdown. Ingestion scripts in src/ingest/ handle the conversion from each source format:

Format Method Notes
EPUB pandoc book.epub -t markdown Preferred — 2–10x more text than PDF scans
PDF (text) pdftotext -layout Poppler. Falls back to OCR if output is garbled.
PDF (scanned) tesseract OCR. Multi-page: convert to images first.
Kindle kindle-ai-export Playwright + GPT-4 vision. Requires Amazon login.
Web article pandoc URL -t markdown Paywalled: paste from reader mode.
YouTube yt-dlp --write-auto-sub VTT → stripped markdown via included script.
Audio / deposition Whisper or AssemblyAI AssemblyAI for speaker diarization.

Each source gets a small .toml config that records author, year, type, and certainty level. The extractor reads these to tag each extracted fact with its provenance.

The two pieces

1 — extract_facts.py

The extraction script. Point it at a markdown file, it scores sentences for factual density, deduplicates near-matches via Jaccard similarity, and appends results to facts.jsonl. Run it once per source. Facts accumulate across runs — 1 document or 1,000, same command.

python3 src/extract_facts.py books/dark-money.md
# → Scanning 6,291 sentences...
# → After dedup: 394 facts. Total in store: 394

Each fact in the output carries: claim text, date, certainty (0–10), status, source IDs, and entity arrays for people, places, and organizations. Certainty is inherited from the source config and adjustable per-fact.

2 — The Claude Code skill

A markdown skill file that teaches Claude Code how to work with the facts database. Drop it into your .claude/skills/ directory and Claude Code can query facts.jsonl directly — by person, date range, organization, tag, or free text — and use the results to answer questions, surface connections, and generate source-linked drafts.

# .claude/skills/research-db.md
# Loaded automatically when Claude Code starts in your project

python3 src/search_facts.py "Charles Koch" --full
python3 src/search_facts.py --person "Charles Koch" --year 2010
python3 src/search_facts.py --person "Charles Koch" --json

When the corpus is stable, migrate to SQLite for relational queries and timeline generation. The migration is safe to re-run at any point as sources accumulate.

FAQ

What's the certainty scale?

0–10, set per source in its config. 8–10: court documents, sworn testimony. 5–7: credible journalism with named sources. 1–4: unverified or single anonymous source. Set conservatively at the source level; raise per-fact when corroborated.

Extraction ran but I got 0 facts.

Three causes: empty input (check wc -l on the markdown file — conversion may have failed silently), score threshold too high (try --threshold 3), or page-marker artifacts from HTML EPUB extraction (strip --- Page N --- lines before running).

pandoc fails with a parseSpine error on my EPUB.

Some EPUBs use raw HTML page files instead of the standard spine structure. Unzip the EPUB and convert the HTML chapters directly — the README has the command.

How do I normalize name variants across sources?

Add aliases to config/entities.toml. The extractor resolves all aliases to the canonical name at extraction time, so searches return consistent results regardless of how a name appears in the source text.

JSONL vs SQLite — when do I migrate?

Stay in JSONL while actively ingesting. Migrate to SQLite when you need entity cross-queries, timeline generation, or theme bucketing. The SQLite database is fully rebuilt from JSONL each time — re-migrating is safe and normal.
