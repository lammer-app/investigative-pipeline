> what is this

A Claude Code skill and Python toolkit for turning research documents — books, articles, depositions, transcripts — into a searchable, queryable evidence database. You point it at a markdown file, it scores sentences for factual density, extracts the keepers into JSONL, and gives you search, timeline, and SQLite query tools on top.

No external dependencies. Python 3.11+ stdlib only. Everything is files you can read, diff, and edit.

The typical workflow: you're deep in a research project with dozens of books, hundreds of articles, and scattered transcripts. You want to find every claim about a specific person across all sources, build a chronological timeline, and surface connections between entities. This does that.

> getting started

Requirements: Python 3.11+. No pip installs needed — everything is stdlib.

Step 1 — Clone and configure paths

# Clone the repo
git clone https://github.com/luxwon/investigative-pipeline
cd investigative-pipeline

# Tell it where your project lives and where to write data
# Option A: export directly
export PIPELINE_VAULT=/Users/you/projects/dark-money
export PIPELINE_ARCHIVE=/Users/you/projects/dark-money/data

# Option B: put both lines in a .env file at your project root
echo 'PIPELINE_VAULT=/Users/you/projects/dark-money' >> /Users/you/projects/dark-money/.env
echo 'PIPELINE_ARCHIVE=/Users/you/projects/dark-money/data' >> /Users/you/projects/dark-money/.env

# Verify paths loaded correctly
python3 src/project_paths.py
# → VAULT:   /Users/you/projects/dark-money
# → ARCHIVE: /Users/you/projects/dark-money/data

# Create the directory structure
mkdir -p $PIPELINE_VAULT/{books,sources,config/sources,profiles}
mkdir -p $PIPELINE_ARCHIVE/{facts,views}

Step 2 — Add a source config

One .toml file per book, article, or document. The filename stem must match the book file (see FAQ). Create it at config/sources/ inside your vault:

# For a book:
# config/sources/dark-money.toml ← filename stem = "dark-money"
# books/dark-money.md            ← book file stem must match

cat > $PIPELINE_VAULT/config/sources/dark-money.toml << 'EOF'
[source]
id             = "SRC-BOOK-mayer-dark-money"
ref            = "Mayer, Jane. Dark Money (2016)"
type           = "book"
title          = "Dark Money"
author         = "Jane Mayer"
year           = 2016
certainty_base = 8
is_memoir      = false
notes          = ""
EOF

Step 3 — Add entity aliases (optional but recommended)

Entities let you normalize name variations so "J. Koch" and "Charles Koch" resolve to the same person in search results.

cat > $PIPELINE_VAULT/config/entities.toml << 'EOF'
[people.koch-charles]
canonical = "Charles Koch"
aliases   = ["C. Koch", "Charles de Ganahl Koch", "Charles G. Koch"]

[people.koch-david]
canonical = "David Koch"
aliases   = ["D. Koch", "David H. Koch"]

[orgs.koch-industries]
canonical = "Koch Industries"
aliases   = ["Koch Industries Inc.", "Koch Ind.", "Koch"]
EOF

Step 4 — Add your source text

# EPUB (recommended — much more text than PDF)
pandoc ~/Downloads/dark-money.epub -t markdown \
  -o $PIPELINE_VAULT/books/dark-money.md

# PDF (lower quality — use when EPUB isn't available)
pdftotext -layout dark-money.pdf $PIPELINE_VAULT/books/dark-money.md

# Article or deposition — goes in sources/, not books/
# (use type = "article" or type = "deposition" in the TOML)
cp my-article.md $PIPELINE_VAULT/sources/dark-money-nyt-2016.md

Step 5 — Extract facts

python3 src/extract_facts.py $PIPELINE_VAULT/books/dark-money.md

# Output:
# [dark-money] Scanning 6,291 sentences...
# [dark-money] Score ≥5: 418 sentences
# [dark-money] After dedup: 394 facts
# → Writing to: /Users/you/projects/dark-money/data/facts/facts.jsonl
# Done. Total facts in store: 394

Step 6 — Build registry and validate

python3 src/build_registry.py
# → 1 source registered → data/facts/sources.jsonl

python3 src/validate.py
# → 394 facts validated, 0 errors, 0 warnings

Step 7 — Search

# Full-text search
python3 src/search_facts.py "Koch brothers" --full

# Filter by person, year, certainty
python3 src/search_facts.py --person "Charles Koch" --year 2010 --certainty 7

# Count facts per tag
python3 src/search_facts.py --tag "financial" --count

# Export to JSON for further processing
python3 src/search_facts.py --person "David Koch" --json > david-koch.json

Step 8 — Migrate to SQLite (after corpus is stable)

python3 src/migrate_to_sqlite.py
# → Migrating 394 facts, 1 source, 47 entities...
# → evidence.db written

# Build a chronological timeline
python3 src/build_timeline.py --full
# → timeline.md written
> how it works 
  books/              sources/              extracts/
  *.md (EPUB→pandoc)  *.md (articles)       *.md (depositions, transcripts)
       │                    │                      │
       └────────────────────┼──────────────────────┘
                            ▼
                  extract_facts.py
                  sentence scoring (FACTUAL_INDICATORS regex)
                  entity alias resolution (config/entities.toml)
                  near-duplicate deduplication (Jaccard)
                            │
                  ┌─────────┴──────────┐
                  ▼                    ▼
           facts.jsonl        sources.jsonl
           media.jsonl        (build_registry.py)
                  │
                  ├── validate.py        schema checks, cross-reference validation
                  ├── generate_views.py  markdown views by year / entity / theme
                  ├── migrate_to_sqlite.py
                  │         │
                  │   evidence.db        13-table relational schema
                  │         │
                  ├── search_facts.py    entity / tag / theme / AI queries
                  └── build_timeline.py  4-agent chronological timeline.md

The extraction engine scores each sentence against FACTUAL_INDICATORS — regex patterns for years, dollar amounts, named entities, action verbs, and attribution phrases. Sentences above the score threshold become fact candidates. Near-duplicates are removed via Jaccard similarity.

The pipeline has two phases. During active research, facts live in JSONL — fast to append, human-readable, easy to grep and diff. When you're ready for richer querying, migrate_to_sqlite.py loads the JSONL into a fully relational SQLite database. You can re-migrate at any time as new sources are added.

> scripts
script what it does when to run it
extract_facts.py Scores sentences, extracts facts to facts.jsonl After adding each new source file
build_registry.py Builds sources.jsonl from config/sources/*.toml After adding or editing a source TOML
validate.py Checks schema integrity and cross-references After any extraction; use --fix to auto-repair
generate_views.py Generates markdown views by year, entity, theme When you want browsable markdown summaries
migrate_to_sqlite.py Loads JSONL into relational SQLite database When corpus is stable; re-run after new extractions
search_facts.py CLI query tool — entity, tag, theme, AI filters Anytime; works from JSONL or SQLite
build_timeline.py Generates chronological timeline.md (4-agent pass) After SQLite migration
project_paths.py Prints resolved paths — run to verify env setup When troubleshooting path issues

All scripts accept --help. All paths read from PIPELINE_VAULT and PIPELINE_ARCHIVE environment variables — or from a .env file at your project root.

> faq

Does the source TOML filename have to match the book filename?

Yes. The stem of the config file determines which book file the extractor uses. config/sources/dark-money.toml maps to books/dark-money.md. If the names don't match, the extractor won't find the source config and will skip metadata enrichment.

Extraction ran but I got 0 facts.

Three common causes:

1. Empty input file. Check: wc -l books/your-source.md. If it's near zero, pandoc or pdftotext failed silently.

2. Score threshold too high. Default is 5. Try lowering it:

python3 src/extract_facts.py books/your-source.md --threshold 3

3. EPUB produced page marker artifacts. If your EPUB was extracted via HTML (not pandoc spine), you may have --- Page N --- blocks that suppress sentence parsing. Strip them first:

python3 -c "
import re, sys
text = open(sys.argv[1]).read()
text = re.sub(r'--- Page \d+ ---\n', '', text)
open(sys.argv[1], 'w').write(text)
" books/your-source.md

pandoc fails with a parseSpine error on my EPUB.

Some EPUBs use raw HTML files instead of the standard spine structure. Extract the EPUB as a zip, find the HTML files, and pandoc from HTML instead:

unzip book.epub -d book_extracted/
# find the HTML content files (usually in OEBPS/ or OPS/)
cat book_extracted/OEBPS/chapter*.html > /tmp/book_combined.html
pandoc /tmp/book_combined.html -t markdown -o books/your-source.md

My source is a PDF, not an EPUB.

pdftotext -layout source.pdf books/your-source.md

PDF text quality varies. Scanned PDFs (images only) require OCR first: tesseract source.pdf out txt then rename out.txt to your target path. EPUB via pandoc gives substantially more text when available — PDF as a fallback only.

I have articles or depositions, not books.

Same pipeline, different directory and source type. Articles go in sources/, transcripts and depositions in extracts/:

# In your source TOML:
type = "article"     # or "deposition", "transcript", "document"

# Run extraction the same way:
python3 src/extract_facts.py $PIPELINE_VAULT/sources/my-article.md

How do I map "J. Smith", "Jane A. Smith", and "Jane Smith" to one entity?

# config/entities.toml
[people.smith-jane]
canonical = "Jane Smith"
aliases   = ["J. Smith", "Jane A. Smith", "Smith, Jane"]

The extractor resolves all aliases to the canonical name in the people array. Search and SQLite queries use canonical names only.

JSONL vs SQLite — when do I need to migrate?

Stay in JSONL while actively ingesting new sources — it's fast to append and easy to inspect. Migrate to SQLite when you want:

  • Entity cross-queries (all facts linking two specific people)
  • Timeline generation (build_timeline.py requires SQLite)
  • Theme/thread bucketing
  • AI-powered analysis via search_facts.py --ai

The SQLite database is fully rebuilt from JSONL each time, so re-migrating after adding new sources is safe and normal.

How do I search for everything involving a specific person?

python3 src/search_facts.py --person "Jane Smith"
python3 src/search_facts.py --person "Jane Smith" --year 2005 --full
python3 src/search_facts.py --person "Jane Smith" --certainty 7 --json > output.json

--person matches against the canonical people array in each fact. If you're getting no results, check that your entities.toml has the right canonical name and that you've re-run extraction after adding it.

How do I add project-specific themes?

Edit the THEMES dict in src/search_facts.py and src/migrate_to_sqlite.py. Both need to match.

THEMES = {
    "financial": "payments, transfers, accounts, donations, funds, investments",
    "legal":     "filed, charged, indicted, deposed, testified, settlement",
    "travel":    "flights, yacht, jet, hotel, trips, visited",
}

Facts are keyword-matched into threads during SQLite migration. Re-run migrate_to_sqlite.py after changing themes.

What does certainty mean?

A 0–10 scale set per source in its TOML (certainty_base), adjustable per fact in facts.jsonl.

  • 8–10 — Court documents, official records, firsthand sworn testimony
  • 5–7 — Credible journalism with named sources, corroborated reporting
  • 1–4 — Unverified, alleged, single anonymous source, advocacy material

Set certainty_base conservatively in the TOML, then raise individual facts when you have corroborating sources.

How do I add a second source and keep the facts separate?

Each source TOML gets its own file. Each fact's sources array carries the source ID. Just add the new TOML, point extraction at the new book file, and re-run build_registry + validate. Facts from different sources accumulate in the same facts.jsonl.

python3 src/extract_facts.py books/second-source.md
python3 src/build_registry.py
python3 src/validate.py
# Total facts in store: 812  (394 + 418)
> data formats

All fact data is newline-delimited JSON (.jsonl). One object per line. Every line is a complete, independent record — you can grep, tail, and sed it directly.

facts.jsonl — one fact per line:

{
  "id":            "F-2016-0042",
  "claim":         "Charles Koch donated $2.5M to the Club for Growth PAC in Q3 2016.",
  "date":          { "value": "2016", "precision": "year" },
  "certainty":     8,
  "status":        "confirmed",
  "review_status": "reviewed",
  "sources": [
    { "source_id": "SRC-BOOK-mayer-dark-money", "page": "142" }
  ],
  "people":        ["Charles Koch"],
  "places":        [],
  "organizations": ["Club for Growth"],
  "tags":          ["financial", "political"],
  "created":       "2025-01-15",
  "modified":      "2025-01-15"
}
id F-{YEAR}-{NNNN}. Year is the event date; NNNN is a zero-padded sequence. Auto-assigned.
claim A single factual statement. Edit manually to clean up extraction artifacts.
date.value ISO format: YYYY-MM-DD, YYYY-MM, or YYYY. Auto-extracted from text.
date.precision exact | month | year | range | undated | approximate
certainty 0–10. Inherited from source certainty_base. Adjust per-fact as needed.
status alleged | confirmed | disputed | disproven. Default: alleged.
sources Array of {source_id, page, quote}. source_id matches the id field in sources.jsonl.
people / places / orgs Entity arrays. Populated by alias resolution. Canonical names from config/entities.toml.
tags Free-form tags. Used for theme bucketing during SQLite migration.
